## Context

The UI needs to show live output as the enrichment run progresses (fetching, matching, writing). A simple request/response model would block until the run completes, giving no feedback during what could be a slow set fetch.

## Decision

Use Server-Sent Events (SSE) for the `/api/run` endpoint. The server spawns the CLI as a Bun subprocess, pipes its stdout/stderr line by line as `data:` SSE events, and sends a final `event: done` with the report JSON when the process exits. The UI subscribes with `EventSource`.

No WebSockets needed — SSE is unidirectional (server → client) which is all that's required. No external dependencies: Bun's `Bun.spawn` and native `ReadableStream` handle everything.

## Consequences

- UI gets live log lines as they happen
- No polling; connection stays open for the duration of the run
- Only one run at a time per browser tab (SSE connection is per-request)
- If the server restarts mid-run the SSE connection drops; the UI shows a reconnect notice
